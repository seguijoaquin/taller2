137 // EventListener class contains a set of call-back functions that will
138 // be called when specific RocksDB event happens such as flush. It can
139 // be used as a building block for developing custom features such as
140 // stats-collector or external compaction algorithm.
141 //
142 // Note that call-back functions should not run for an extended period of
143 // time before the function returns, otherwise RocksDB may be blocked.
144 // For example, it is not suggested to do DB::CompactFiles() (as it may
145 // run for a long while) or issue many of DB::Put() (as Put may be blocked
146 // in certain cases) in the same thread in the EventListener callback.
147 // However, doing DB::CompactFiles() and DB::Put() in another thread is
148 // considered safe.
149 //
150 // [Threading] All EventListener callback will be called using the
151 // actual thread that involves in that specific event. For example, it
152 // is the RocksDB background flush thread that does the actual flush to
153 // call EventListener::OnFlushCompleted().
154 //
155 // [Locking] All EventListener callbacks are designed to be called without
156 // the current thread holding any DB mutex. This is to prevent potential
157 // deadlock and performance issue when using EventListener callback
158 // in a complex way. However, all EventListener call-back functions
159 // should not run for an extended period of time before the function
160 // returns, otherwise RocksDB may be blocked. For example, it is not
161 // suggested to do DB::CompactFiles() (as it may run for a long while)
162 // or issue many of DB::Put() (as Put may be blocked in certain cases)
163 // in the same thread in the EventListener callback. However, doing
164 // DB::CompactFiles() and DB::Put() in a thread other than the
165 // EventListener callback thread is considered safe.
166 class EventListener {
167  public:
168  // A call-back function to RocksDB which will be called whenever a
169  // registered RocksDB flushes a file. The default implementation is
170  // no-op.
171  //
172  // Note that the this function must be implemented in a way such that
173  // it should not run for an extended period of time before the function
174  // returns. Otherwise, RocksDB may be blocked.
175  virtual void OnFlushCompleted(DB* /*db*/,
176  const FlushJobInfo& /*flush_job_info*/) {}
177 
178  // A call-back function for RocksDB which will be called whenever
179  // a SST file is deleted. Different from OnCompactionCompleted and
180  // OnFlushCompleted, this call-back is designed for external logging
181  // service and thus only provide string parameters instead
182  // of a pointer to DB. Applications that build logic basic based
183  // on file creations and deletions is suggested to implement
184  // OnFlushCompleted and OnCompactionCompleted.
185  //
186  // Note that if applications would like to use the passed reference
187  // outside this function call, they should make copies from the
188  // returned value.
189  virtual void OnTableFileDeleted(const TableFileDeletionInfo& /*info*/) {}
190 
191  // A call-back function for RocksDB which will be called whenever
192  // a registered RocksDB compacts a file. The default implementation
193  // is a no-op.
194  //
195  // Note that this function must be implemented in a way such that
196  // it should not run for an extended period of time before the function
197  // returns. Otherwise, RocksDB may be blocked.
198  //
199  // @param db a pointer to the rocksdb instance which just compacted
200  // a file.
201  // @param ci a reference to a CompactionJobInfo struct. 'ci' is released
202  // after this function is returned, and must be copied if it is needed
203  // outside of this function.
204  virtual void OnCompactionCompleted(DB* /*db*/,
205  const CompactionJobInfo& /*ci*/) {}
206 
207  // A call-back function for RocksDB which will be called whenever
208  // a SST file is created. Different from OnCompactionCompleted and
209  // OnFlushCompleted, this call-back is designed for external logging
210  // service and thus only provide string parameters instead
211  // of a pointer to DB. Applications that build logic basic based
212  // on file creations and deletions is suggested to implement
213  // OnFlushCompleted and OnCompactionCompleted.
214  //
215  // Note that if applications would like to use the passed reference
216  // outside this function call, they should make copies from these
217  // returned value.
218  virtual void OnTableFileCreated(const TableFileCreationInfo& /*info*/) {}
219 
220  virtual ~EventListener() {}
221 };
